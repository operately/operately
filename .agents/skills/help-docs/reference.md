# Help Docs Reference

This document expands on [SKILL.md](SKILL.md). Use it for git recipes, filtering
rules, code research paths, and what to capture per documentation candidate.

## Existing help pages

Published help content lives in the sibling `operately-website` repo at
`src/content/docs/help/`. Sidebar sections and slugs are defined in
`src/config/helpCenter.js`.

When the sibling repo is available at `../operately-website/`, scan
`helpCenter.js` and existing MDX slugs to decide between `new_page` and
`update_page` and name the closest existing page.

## Git commands

Replace `{baseline}` with the resolved SHA or tag.

### Validate baseline and show range

```bash
git rev-parse --verify {baseline}^{commit}
git rev-parse HEAD
git rev-list --count {baseline}..HEAD
git log -1 --format='%h %s (%ci)' {baseline}
```

### List merge commits with PR titles

Merge commits usually carry the PR title (with `feat:` / `fix:` prefix):

```bash
git log {baseline}..HEAD --merges --oneline
git log {baseline}..HEAD --merges --format='%h %s'
```

### Path-filtered history (user-facing surfaces)

```bash
git log {baseline}..HEAD --oneline -- \
  app/assets/js/pages/ \
  app/assets/js/features/ \
  app/assets/js/routes/ \
  app/assets/js/layouts/ \
  turboui/src/

git diff --stat {baseline}..HEAD -- \
  app/assets/js/pages/ \
  app/assets/js/features/ \
  app/assets/js/routes/ \
  turboui/src/
```

### Per-commit changed files

```bash
git log {baseline}..HEAD --name-only --pretty=format:'COMMIT %h %s'
```

### Optional: weekly merged PR list

For release-window context (requires `gh` CLI):

```bash
./scripts/changelog.sh --weeks-ago=1
```

This is supplementary — baseline..HEAD is the primary source.

## Path watchlist

### Primary (user-facing UI)

| Path | What to find |
| ---- | ------------ |
| `app/assets/js/pages/` | New screens, wizards, settings, page layout |
| `app/assets/js/features/` | Feature-specific UI, modals, forms, activities |
| `turboui/src/` | Shared components; new `*Page` or task/billing UI |
| `app/assets/js/routes/paths.tsx` | New, changed, or removed URLs |
| `app/assets/js/routes/index.tsx` | Route registration |
| `app/assets/js/pages/index.tsx` | Page exports |
| `app/assets/js/layouts/` | Top nav, sidebar, dropdown changes |
| `app/assets/js/models/` | New user actions wired to UI |
| `ee/assets/js/` | Enterprise/admin UI |

### Secondary (validation and intent)

| Path | Use |
| ---- | --- |
| `app/test/features/` | Confirm user flows and step order |
| `specs/` | Product intent for major features |
| `app/assets/js/pages/**/*.test.tsx` | Expected labels and behavior |

### API surface (flag only — no manual help page)

| Path | Use |
| ---- | --- |
| `app/lib/operately_web/api.ex` | New/changed external endpoints |
| `app/lib/operately_web/api/` | Handler inputs/outputs |

External API docs are auto-generated by `Operately.ApiDocs.Generator` and
synced to the website by CI. Flag changes as `api_auto_sync`.

### Ignore unless locating UI entry

| Path | Why |
| ---- | --- |
| `app/lib/operately/` (operations, schemas) | Business logic — use only to find which page renders a feature |
| `app/priv/repo/migrations/` | Schema changes rarely need help docs alone |
| `app/lib/operately/data/change_*.ex` | Data backfills |
| `app/test/` (non-feature) | Unit/integration tests |
| `scripts/`, `.semaphore/`, `.github/` | CI and tooling |
| `Makefile`, dependency files | Infrastructure |

Activity backend (`app/lib/operately/activities/`, serializers) is usually
internal unless feed copy or a new user-visible action changed.

## Filter heuristics

### Include (likely doc candidate)

1. PR title starts with `feat:` or `fix:` and describes UI, permissions,
   navigation, or user-visible copy.
2. Diff touches primary watchlist paths.
3. New page directory under `app/assets/js/pages/`.
4. New route in `paths.tsx`.
5. New button, menu item, settings panel, wizard step, or blocked-state page.
6. Feature tests added for a new user flow.
7. Spec updated in `specs/` for a shipped product area.
8. Navigation removed or renamed (existing help pages may be wrong).

### Exclude (skip with reason)

1. PR title `chore:` — internal per project guidelines.
2. Diff limited to tests, CI, scripts, deps, or refactors with no behavior change.
3. Backend-only changes with no matching frontend/turboui diff.
4. Data migrations without UI label or flow changes.
5. Activity plumbing when feed text and user actions are unchanged.
6. `docs:` commits in operately — developer docs, not product help.
7. Email-only or background job changes with no new in-app steps.

### Gray area (research before deciding)

| Change | Guidance |
| ------ | -------- |
| `fix:` copy-only | `update_page` only if help pages quote the old labels |
| New activity feed item | Usually no new how-to; update overview if feed behavior changed |
| Permission enforcement fix | `update_page` on permissions reference if access rules changed |
| TurboUI-only component | User-facing if it changes labels/flows in app pages |

## Code research map

For each doc candidate cluster, read source until you can describe the user flow.

### What to extract

- Top navigation items, sidebar links, tab names
- Button labels, dropdown options, dialog titles, menu items
- Form field labels, placeholders, validation messages
- Step order in multi-step dialogs and wizards
- Permission gates, role checks, access level names shown in UI
- Empty state text, success toasts, error messages
- Multiple entry paths to the same action (nav vs context menu vs inline)
- Removed routes or deprecated screens (flag `update_page` on related help)

### What to ignore

- Elixir operations, Ecto schemas, GraphQL resolver internals
- Database migrations and data models
- Activity content handlers and serializers (unless feed copy changed)
- API request/response shapes (unless flagging `api_auto_sync`)

Use backend code only to locate which frontend page renders a feature, then
switch to reading the frontend source.

### Research workflow

1. Start from the merge commit's changed files in primary watchlist paths.
2. Read the page component and its children for labels and flow order.
3. Check `turboui/src/` when the page imports shared components.
4. Read matching feature tests to confirm step order and assertions.
5. Check `specs/` if the feature area has a product spec.
6. If `operately-website` is available, search `src/content/docs/help/` for
   related slugs and read the closest existing page title/slug.

## Doc action types

| Action | Meaning |
| ------ | ------- |
| `new_page` | New help page needed; suggest slug and sidebar section |
| `update_page` | Revise an existing page; cite `/help/{slug}` |
| `api_auto_sync` | No manual page — CI syncs API reference after deploy |
| `skip` | No doc work; state reason |

### Help center sections

When suggesting `new_page`, pick the closest section from the website sidebar.
Common sections: Meet Operately, Account management, Global Search, User profiles,
Work maps, Permissions & Access, Goal tracking, Project management, Spaces,
Kanban board, Discussions, Documents & Files, Review, People, Company
administration, Notifications, Self-hosted installations, CLI.

Read `operately-website/src/config/helpCenter.js` for the authoritative list when
available.

### Page types

| Type | When |
| ---- | ---- |
| How-to | User performs one specific task |
| Overview | Explain a product area or concept |
| Reference | Permissions, settings, access matrices |

## What to capture

For each documentation candidate, gather:

- Doc action (`new_page`, `update_page`, `api_auto_sync`, or `skip`)
- Suggested slug and help center section (for `new_page`)
- Closest existing page at `/help/{slug}` (for `update_page`)
- Source commits and PR titles
- Changed file paths
- Change summary (1–3 terse sentences)
- User capability (what users can now do)
- Entry points (exact nav/menu paths from code)
- UI labels (quoted exactly from source)
- Flow (step order inferred from code)
- Permissions and roles (from UI gates; note if unknown)
- Edge cases (empty states, errors, blocked flows)
- Page type (how-to, overview, or reference)
- Related help pages that should cross-link

For `api_auto_sync` items, note the namespace or endpoint changes.

For skipped commits, note the PR title and reason for exclusion.

## Example

For `feat: add configurable task reminders` (#4616):

- **Doc action:** `new_page`
- **Suggested slug:** `configure-task-reminders`
- **Suggested section:** Kanban board (also relevant: Project management)
- **Existing page:** none found
- **Source:** f0239cc29 feat: add configurable task reminders (#4616)
- **Changed files:** `turboui/src/TaskPage/Sidebar.tsx`, task views in kanban/project/space pages
- **Change summary:** Tasks support configurable reminder schedules from multiple task views.
- **User capability:** Set when and how reminders fire on a task.
- **Entry points:** Task detail sidebar, Kanban card, milestone task list, project task list, space task views
- **UI labels:** sidebar section "Reminders"; button "Add reminder"; types "Before due date", "Due date", "Overdue", "On date"
- **Flow:** Open a task with assignees → find Reminders section → click Add reminder → choose type and options
- **Permissions / roles:** editable when `canEdit` is true; section hidden when task has no assignees
- **Edge cases:** default reminder type depends on whether task has a due date
- **Page type:** how-to
- **Cross-links:** kanban and project task pages

## Common mistakes to avoid

- Paraphrasing UI labels instead of quoting source
- Grouping unrelated features into one item — cluster by area
- Documenting backend modules, GraphQL, or database schema
- Treating API endpoint changes as manual help page work — use `api_auto_sync`
- Omitting reasons for excluded commits
- Forgetting to validate the baseline SHA/tag before starting
